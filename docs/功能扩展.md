# 功能扩展

## 1. 扩展方向

本项目后续扩展方向确定为：

```text
工位危险区域人员入侵 + 异常停留 + 安全帽/反光衣合规检测扩展
```

该方向适合突出项目中的图像采集、边缘 AI 推理、规则判断、告警联动、数据记录和 Web 展示能力。相比火焰、烟雾等更适合传感器检测的场景，本扩展方向更能体现 YOLOv8 视觉检测的价值。

## 2. 应用场景

项目可包装为：

```text
基于 ESP32-S3-CAM 与 YOLOv8 的工位安全合规边缘监测系统
```

典型应用场景：

- 小型工位设备区域。
- 机械设备危险区域。
- 无人值守设备间。
- 教学实验工位或实训工位。
- 简易工业安全合规演示场景。

系统通过 ESP32-S3-CAM 周期采集工位图像并上传到 Linux 边缘节点，Linux 端使用 YOLOv8 完成目标检测，并结合危险区域、连续触发和告警抑制规则判断是否产生安全事件。

## 3. 当前基础

当前项目已经具备以下基础能力：

- ESP32-S3-CAM 摄像头采集 JPEG 图片。
- ESP32 Wi-Fi 联网和 HTTP 图片上传。
- Linux FastAPI 接收图片。
- Linux 端 YOLOv8 目标检测。
- OpenCV 绘制检测框并保存结果图。
- SQLite 保存检测事件、设备状态和目标明细。
- Web 页面展示最新检测、设备状态和历史告警。
- ESP32 根据 `alarm=true` 触发板载 LED 告警。

当前不足：

- 告警规则只判断 `person`。
- 未支持危险区域 ROI 判断。
- 未支持人员连续停留判断。
- 未支持告警抑制。
- 未支持安全帽、反光衣等合规检测类别。
- Web 页面还未突出工位安全事件类型和告警等级。

## 4. 功能目标

### 4.1 第一阶段：危险区域人员入侵检测

检测目标：

```text
person
```

规则：

```text
当 person 检测框中心点落入危险区域 ROI 时，触发危险区域入侵告警。
```

事件类型：

```text
danger_zone_intrusion
```

告警等级：

```text
high
```

该阶段不需要更换模型，普通 YOLOv8n 的 COCO 预训练模型即可检测 `person`。

### 4.2 第二阶段：人员异常停留检测

检测目标：

```text
person
```

规则：

```text
同一设备连续多次检测到 person 出现在危险区域 ROI 内，判定为异常停留。
```

示例：

```text
连续 1 次命中危险区域 -> suspected
连续 3 次命中危险区域 -> alarm
```

事件类型：

```text
abnormal_stay
```

告警等级：

```text
medium 或 high
```

该阶段利用 ESP32 周期性上传图片的特点，不需要视频流，也不需要复杂目标跟踪。

### 4.3 第三阶段：告警抑制

目标：

```text
避免同一设备在短时间内重复产生大量相同告警。
```

规则：

```text
同一 device_id、同一 event_type 在 ALARM_SUPPRESS_SECONDS 时间内只触发一次高等级告警。
```

示例：

```text
10 秒内多次检测到人员停留，只记录状态，不重复触发 ESP32 LED 告警。
```

### 4.4 第四阶段：安全帽/反光衣合规检测扩展

检测目标：

```text
person
helmet
vest
no_helmet
no_vest
```

推荐规则：

```text
检测到 person 且检测到 no_helmet -> 安全帽违规告警
检测到 person 且检测到 no_vest -> 反光衣违规告警
检测到 person 进入危险区域且未佩戴安全帽 -> 高等级告警
```

事件类型：

```text
no_helmet
no_vest
safety_violation
```

告警等级：

```text
no_helmet -> high
no_vest -> medium
person in danger zone + no_helmet -> high
```

注意：

- 普通 `yolov8n.pt` 不包含 `helmet`、`vest`、`no_helmet`、`no_vest` 类别。
- 该阶段需要准备安全帽/反光衣检测模型，或使用已有自定义 YOLOv8 权重。
- 在没有自定义模型前，代码可以先完成规则框架，实际运行时只启用 `person` 相关规则。

## 5. 具体实现方案

### 5.1 配置扩展

修改文件：

```text
linux/config.yaml
linux/app/core/config.py
```

新增配置建议：

```yaml
DANGER_ZONE_ENABLED: true
DANGER_ZONE_ROI:
  - [0.25, 0.45]
  - [0.95, 0.45]
  - [0.95, 0.95]
  - [0.25, 0.95]
ALARM_TRIGGER_COUNT: 3
ALARM_SUPPRESS_SECONDS: 10
ALARM_CLASSES:
  person:
    event_type: danger_zone_intrusion
    alarm_level: high
  no_helmet:
    event_type: no_helmet
    alarm_level: high
  no_vest:
    event_type: no_vest
    alarm_level: medium
```

说明：

- `DANGER_ZONE_ROI` 使用归一化坐标，范围为 `0.0` 到 `1.0`。
- 使用归一化坐标后，不依赖固定图片分辨率。
- `ALARM_TRIGGER_COUNT` 用于连续触发。
- `ALARM_SUPPRESS_SECONDS` 用于告警抑制。

### 5.2 ROI 判断模块

建议新增文件：

```text
linux/app/rules/roi_utils.py
```

核心职责：

- 计算检测框中心点。
- 将像素坐标转换为归一化坐标。
- 判断点是否在危险区域多边形内。

核心函数建议：

```python
def box_center(box: list[int]) -> tuple[float, float]:
    """计算检测框中心点。"""

def normalize_point(x: float, y: float, width: int, height: int) -> tuple[float, float]:
    """将像素坐标转换为归一化坐标。"""

def point_in_polygon(point: tuple[float, float], polygon: list[list[float]]) -> bool:
    """判断点是否位于 ROI 多边形内。"""
```

实现原则：

- 不依赖大型第三方库。
- 保持规则模块轻量。
- 单元测试覆盖边界点、区域内、区域外三种情况。

### 5.3 告警规则引擎扩展

修改文件：

```text
linux/app/rules/alarm_rule_engine.py
```

当前规则：

```text
检测到 person 且置信度超过阈值 -> alarm=true
```

扩展后规则：

```text
1. 过滤置信度低于阈值的目标。
2. 判断 person 是否进入危险区域 ROI。
3. 统计同一设备连续命中次数。
4. 连续次数未达阈值时返回 suspected。
5. 连续次数达到阈值后返回 alarm。
6. 告警抑制时间内不重复触发高等级告警。
7. 如果模型支持 no_helmet/no_vest，则额外判断安全合规事件。
```

输出结构保持兼容：

```python
{
    "alarm": True,
    "alarm_level": "high",
    "alarm_action": "beep_led",
    "alarm_status": "alarm",
    "event_type": "danger_zone_intrusion",
}
```

为兼容现有 ESP32 逻辑，必须继续返回：

```text
alarm
alarm_level
alarm_action
event_type
```

### 5.4 检测结果增强

当前目标结构：

```python
{
    "class": "person",
    "confidence": 0.86,
    "box": [x1, y1, x2, y2],
}
```

建议扩展内部使用字段：

```python
{
    "class": "person",
    "confidence": 0.86,
    "box": [x1, y1, x2, y2],
    "in_danger_zone": True,
}
```

注意：

- 数据库存储仍保持现有 `detection_targets` 表结构。
- 如果后续需要保存 ROI 命中状态，可新增字段或在事件详情中扩展。
- 初期不要大改数据库，优先保持现有结构稳定。

### 5.5 结果图绘制增强

修改文件：

```text
linux/app/services/image_service.py
```

建议增强：

- 在结果图上绘制危险区域 ROI。
- 对进入危险区域的 `person` 使用红色框。
- 普通检测目标使用绿色框。
- 标签显示类别、置信度和事件状态。

示例显示：

```text
person 0.82 danger_zone
no_helmet 0.91 violation
```

实现原则：

- OpenCV 只负责绘制，不负责告警规则。
- ROI 绘制失败不影响上传和检测主链路。

### 5.6 Web 页面增强

修改文件：

```text
linux/app/api/web_api.py
```

建议增强内容：

- 最新事件显示 `event_type`。
- 告警列表显示告警等级。
- 支持按 `event_type` 筛选。
- 支持按 `alarm_status` 筛选。
- 首页显示今日告警数量。
- 首页显示最近一次高等级告警。

页面展示建议：

```text
设备状态
最新检测图片
最新事件类型
告警等级
目标类别
是否已处理
历史告警列表
```

不建议当前阶段加入复杂前端框架，保持轻量 HTML 页面即可。

### 5.7 数据库扩展

当前数据库已经包含：

```text
alarm_events.event_type
alarm_events.alarm_level
alarm_events.alarm_status
detection_targets.target_class
detection_targets.confidence
detection_targets.box_x1/box_y1/box_x2/box_y2
```

第一阶段不需要修改数据库结构。

后续如需更完整追踪 ROI 和合规状态，可考虑新增字段：

```text
detection_targets.in_danger_zone
alarm_events.rule_name
alarm_events.suppressed
```

但当前建议先不改表结构，避免扩大影响范围。

### 5.8 ESP32 端影响

ESP32 端暂时不需要大改。

原因：

- ESP32 当前只关心 Linux 返回的 `alarm=true/false`。
- 告警类型和等级由 Linux 端判断。
- ESP32 端 LED 闪烁逻辑可以继续复用。

后续可选扩展：

- 根据 `alarm_level` 调整 LED 闪烁频率。
- 根据 `alarm_action` 区分 LED、蜂鸣器或舵机动作。
- 接入蜂鸣器后，高等级告警启用声音提示。

当前阶段建议保持 ESP32 稳定，不优先修改。

## 6. 推荐开发顺序

### 步骤一：配置化 ROI

目标：

```text
支持在配置文件中定义危险区域。
```

需要修改：

- `linux/config.yaml`
- `linux/app/core/config.py`

验收标准：

- 服务端能读取 ROI 配置。
- ROI 配置为空时系统仍正常运行。

### 步骤二：实现 ROI 判断工具

目标：

```text
判断检测目标是否进入危险区域。
```

需要新增：

- `linux/app/rules/roi_utils.py`
- `linux/tests/test_roi_utils.py`

验收标准：

- 点在 ROI 内返回 `True`。
- 点在 ROI 外返回 `False`。
- 边界输入不导致异常。

### 步骤三：扩展告警规则

目标：

```text
从“检测到人就告警”升级为“人进入危险区域才告警”。
```

需要修改：

- `linux/app/rules/alarm_rule_engine.py`
- `linux/tests/test_detection.py` 或新增 `linux/tests/test_alarm_rule_engine.py`

验收标准：

- `person` 在 ROI 外不触发告警。
- `person` 在 ROI 内触发危险区域入侵告警。
- 返回 `event_type=danger_zone_intrusion`。

### 步骤四：实现连续触发

目标：

```text
连续多次检测到危险区域人员才进入 alarm 状态。
```

需要修改：

- `linux/app/rules/alarm_rule_engine.py`

验收标准：

- 第一次命中返回 `suspected`。
- 达到配置次数后返回 `alarm`。
- 未命中后计数可以重置或衰减。

### 步骤五：实现告警抑制

目标：

```text
短时间内不重复触发相同告警。
```

需要修改：

- `linux/app/rules/alarm_rule_engine.py`

验收标准：

- 抑制时间内重复命中不重复触发高等级告警。
- 抑制时间结束后可再次触发告警。

### 步骤六：绘制危险区域和事件标记

目标：

```text
结果图片能直观看到危险区域和违规目标。
```

需要修改：

- `linux/app/services/image_service.py`

验收标准：

- 结果图显示 ROI 边框。
- 进入危险区域的目标使用醒目颜色。

### 步骤七：Web 页面展示增强

目标：

```text
Web 页面能展示工位安全事件信息。
```

需要修改：

- `linux/app/api/web_api.py`
- 查询接口相关仓储或服务模块。

验收标准：

- 页面显示事件类型。
- 页面显示告警等级。
- 页面可以查看历史违规事件。

### 步骤八：安全帽/反光衣模型扩展

目标：

```text
通过替换 YOLOv8 自定义权重，实现安全帽和反光衣合规检测。
```

需要准备：

- 支持 `helmet`、`vest`、`no_helmet`、`no_vest` 的 YOLOv8 模型。
- 将模型放入 `linux/models/`。
- 修改 `linux/config.yaml` 中的 `MODEL_PATH`。

验收标准：

- 检测结果中出现安全帽或反光衣相关类别。
- 告警规则能根据类别输出对应事件类型。
- Web 页面能展示合规检测事件。

## 7. 推荐最终功能组合

建议最终展示功能为：

```text
1. ESP32-S3-CAM 周期采集工位图像。
2. Linux 端 YOLOv8 检测人员。
3. 系统判断人员是否进入危险区域 ROI。
4. 连续多次命中后触发异常停留或危险区域入侵告警。
5. 告警事件写入 SQLite。
6. Web 页面展示原图、结果图、事件类型、告警等级和处理状态。
7. ESP32 收到 alarm=true 后执行板载 LED 告警。
8. 后续可替换安全帽/反光衣模型扩展合规检测。
```

## 8. 风险点

### 8.1 模型类别限制

普通 `yolov8n.pt` 可用于人员检测，但不支持安全帽和反光衣类别。

应对：

- 第一阶段先完成 `person + ROI + 停留规则`。
- 安全帽/反光衣作为自定义模型扩展。

### 8.2 摄像头视角限制

ESP32-S3-CAM 画质和视角有限，摄像头需要固定安装，保证危险区域在画面中清晰可见。

应对：

- 使用固定摄像头位置。
- ROI 使用归一化坐标。
- 优先检测较大的人员目标。

### 8.3 误报问题

单帧检测可能误报。

应对：

- 使用连续触发规则。
- 使用告警抑制。
- 提高置信度阈值。

### 8.4 工业合规模型获取

安全帽和反光衣检测需要自定义模型或现成权重。

应对：

- 将该功能作为扩展能力。
- 当前代码先预留配置化规则和类别判断。

### 8.5 ESP32 固件容量

当前 ESP32 app 分区剩余空间较少，后续不建议在 ESP32 端增加复杂逻辑。

应对：

- 主要扩展放在 Linux 端。
- ESP32 端保持采集、上传和本地告警职责。

## 9. 简历表达建议

可以将项目描述为：

```text
基于 ESP32-S3-CAM 与 YOLOv8 设计工位安全合规边缘监测系统，ESP32 端负责图像采集、Wi-Fi 上传和本地 LED 告警，Linux 边缘端基于 FastAPI 接收图片并调用 YOLOv8 完成人员检测。系统结合危险区域 ROI、连续触发和告警抑制规则，实现工位危险区域人员入侵和异常停留检测，并通过 SQLite 和 Web 页面完成告警事件追踪与可视化展示。系统支持通过替换自定义 YOLOv8 权重扩展安全帽、反光衣等工业安全合规检测能力。
```

项目亮点可写为：

- 端边协同架构：ESP32 负责采集与联动，Linux 负责 AI 推理和数据管理。
- 规则引擎扩展：支持危险区域 ROI、连续触发和告警抑制。
- 工程闭环完整：图片上传、模型检测、告警判断、数据库记录、Web 展示、本地联动。
- 可扩展模型设计：通过替换 YOLOv8 权重支持安全帽、反光衣等合规检测。
