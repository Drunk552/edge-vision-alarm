# 项目优化方案

## 1. 优化目标

本项目当前已经完成 ESP32-S3-CAM 图像采集、HTTP 图片上传、Linux YOLOv8 推理、SQLite 数据存储、Web 页面展示和本地 LED 告警的基础闭环。后续优化的重点不放在告警规则增强和 AI 模型效果提升上，而是面向工程化、可扩展性和简历项目表达能力，将系统从“单设备可运行 Demo”优化为“支持多设备接入、稳定运行、可观测、可维护的边缘视觉告警系统”。

优化目标如下：

- 支持多个 ESP32-S3-CAM 设备并发接入。
- 提升 Linux 服务端在多设备上传场景下的稳定性。
- 优化 ESP32 端网络异常、上传失败和资源紧张时的处理能力。
- 完善设备管理、接口鉴权、图片存储和系统日志。
- 增强 Web 管理页面的设备与事件查看能力。
- 保持 HTTP 图片上传主链路，避免为了复杂度而引入不必要的裸 TCP 主链路。

## 2. 当前系统现状

当前系统数据流如下：

```text
ESP32-S3-CAM 拍照
-> Wi-Fi 上传 JPEG 图片
-> Linux FastAPI 接收图片
-> YOLOv8 目标检测
-> 告警状态返回
-> SQLite 保存事件、设备状态和检测目标
-> Web 页面展示
-> ESP32 根据 alarm 字段触发 LED 告警
```

当前 Linux 端已经具备以下能力：

- `POST /api/upload`：接收 ESP32 上传的 JPEG 图片。
- `POST /api/heartbeat`：接收设备心跳。
- `GET /api/health`：服务健康检查。
- `GET /api/latest`：查询最新事件。
- `GET /api/alarms`：分页查询历史事件。
- `GET /api/devices`：分页查询设备状态。
- `POST /api/alarms/{event_id}/handle`：标记事件已处理。
- `GET /`：Web 管理页面。

当前 ESP32 端已经具备以下能力：

- Wi-Fi STA 联网与断线重连。
- 周期性心跳上报。
- OV3660 摄像头初始化与 JPEG 图片采集。
- `multipart/form-data` 图片上传。
- 解析服务端返回的 `alarm` 字段。
- 使用 GPIO38 板载 LED 执行本地告警。

当前主要可优化点：

- 上传接口与 YOLO 推理同步执行，多设备同时上传时容易被推理耗时阻塞。
- 多设备虽然已有 `device_id`，但设备管理信息还不够完整。
- ESP32 上传响应解析方式较简单，目前主要依赖字符串匹配。
- Linux 端配置了 `API_TOKEN`，ESP32 也支持 `X-API-Token`，但服务端鉴权能力还可以补齐。
- 图片长期保存策略尚不完善，多设备运行后可能导致磁盘占用增长。
- Web 页面目前是最小可用版本，后续可增强设备视角和事件管理能力。

## 3. 通信方案优化建议

### 3.1 保留 HTTP 作为图片上传主链路

当前项目使用 HTTP 上传 JPEG 图片是合理的工程选择。ESP32-S3-CAM 的主要数据是周期性图片，HTTP 具备以下优势：

- ESP-IDF 官方 HTTP Client 支持成熟。
- FastAPI 对文件上传、表单字段和 JSON 响应支持完善。
- 每次上传请求相互独立，便于失败重试和问题定位。
- 多设备接入时可通过 `device_id` 区分数据来源。
- 与 Web API、健康检查、查询接口保持统一技术栈。

不建议将图片上传主链路直接替换为裸 TCP。裸 TCP 需要自行处理协议设计、粘包拆包、心跳、重连、超时、鉴权和并发连接管理，会显著增加复杂度。对于 JPEG 图片上传场景，HTTP 头部开销通常不是主要瓶颈，真正的瓶颈更可能来自网络质量、图片大小、YOLO 推理耗时和数据库写入。

### 3.2 可选扩展控制通道

如果后续希望体现更丰富的网络技术栈，可以采用“数据面与控制面分离”的方式：

```text
HTTP：图片上传数据面
HTTP 心跳 / MQTT / WebSocket / TCP：设备状态与控制面
```

推荐优先级：

| 方案 | 适用场景 | 建议 |
| --- | --- | --- |
| HTTP 心跳 + HTTP 响应控制 | 当前阶段最稳妥 | 推荐保留 |
| MQTT | IoT 多设备状态与命令下发 | 后续可扩展 |
| WebSocket | Web 页面实时刷新、服务端推送 | 后续可扩展 |
| 裸 TCP 长连接 | 自定义协议、连接管理能力展示 | 作为扩展实验，不建议替换主链路 |

简历表达建议：

```text
采用 HTTP 作为图片上传数据面，预留 MQTT/WebSocket 控制面扩展，实现设备状态上报、服务端控制指令下发和多设备统一管理。
```

## 4. 多设备接入优化

### 4.1 设备状态字段增强

当前系统已有 `device_id`、`rssi`、`free_heap`、`status`、`last_seen` 等基础字段。后续可扩展以下字段：

| 字段 | 说明 |
| --- | --- |
| device_ip | 设备最近一次请求 IP |
| firmware_version | ESP32 固件版本 |
| camera_status | 摄像头状态 |
| upload_count | 累计上传次数 |
| alarm_count | 累计告警次数 |
| last_upload_at | 最近一次图片上传时间 |
| last_heartbeat_at | 最近一次心跳时间 |
| online | 根据最近心跳时间计算的在线状态 |

优化后的设备状态可以支持 Web 页面展示：

```text
设备 ID / 在线状态 / IP / RSSI / 剩余堆内存 / 最近心跳 / 最近上传 / 上传次数 / 告警次数
```

### 4.2 设备在线离线判断

当前设备状态主要依赖 ESP32 主动上报。后续可在服务端增加在线状态判断逻辑：

```text
当前时间 - last_seen <= 在线超时时间 -> online
当前时间 - last_seen > 在线超时时间 -> offline
```

建议配置项：

```yaml
DEVICE_OFFLINE_SECONDS: 30
```

这样即使 ESP32 没有主动上报离线，Linux 端也能通过心跳超时判断设备是否离线。

### 4.3 多设备数据隔离

建议所有查询接口都支持 `device_id` 过滤：

```text
GET /api/latest?device_id=esp32_s3_cam_001
GET /api/alarms?device_id=esp32_s3_cam_001
GET /api/devices?device_id=esp32_s3_cam_001
```

Web 页面也应支持按设备筛选最新图片、历史事件和设备状态，方便多摄像头部署后定位具体设备。

## 5. 推理任务队列优化

### 5.1 当前同步推理问题

当前 `/api/upload` 的处理流程是：

```text
接收图片
-> 保存图片
-> 立即执行 YOLOv8 推理
-> 保存检测结果
-> 写数据库
-> 返回响应
```

单设备低频上传时该方式简单可靠。但多设备同时上传时，YOLO 推理会阻塞 HTTP 请求，可能导致：

- ESP32 等待响应时间过长。
- 多设备上传请求排队。
- 服务端响应变慢。
- ESP32 端 HTTP 超时概率增加。

### 5.2 建议改造方向

建议引入轻量级后台推理任务队列：

```text
HTTP 接收图片
-> 保存原始图片
-> 创建 pending 事件
-> 将 event_id 和图片路径放入推理队列
-> 立即返回 upload accepted
-> 后台 worker 执行 YOLO 推理
-> 更新事件状态、结果图和检测目标
```

事件状态建议增加：

| 状态 | 说明 |
| --- | --- |
| pending | 图片已接收，等待推理 |
| processing | 正在推理 |
| done | 推理完成 |
| failed | 推理失败 |

数据库建议增加字段：

```text
process_status
inference_time_ms
error_message
updated_at
```

### 5.3 队列实现选择

当前项目可以先使用 Python 内置能力实现轻量队列：

```text
asyncio.Queue / queue.Queue + 后台 worker
```

后续如果项目规模扩大，可扩展为：

```text
Redis Queue / Celery / multiprocessing worker
```

当前阶段不建议一开始引入 Redis 或 Celery，避免课程设计项目依赖过重。

### 5.4 对 ESP32 响应方式的影响

如果上传接口改成异步推理，ESP32 可能无法在本次上传响应中立即得到最终告警结果。可选方案：

方案一：保留同步推理，用于当前闭环演示。

方案二：异步推理后，ESP32 通过心跳接口轮询最近告警状态。

方案三：后续增加 MQTT/WebSocket/TCP 控制通道，由服务端主动下发告警。

建议当前实现路径：

```text
先保留同步推理主链路
-> 增加可配置的异步推理模式
-> 简历中体现接入层与推理层可解耦设计
```

## 6. ESP32 端稳定性优化

### 6.1 上传失败处理

当前 ESP32 上传失败后会等待下一个周期继续上传。后续可以增加失败状态管理：

```text
连续失败次数
最近一次失败原因
服务端不可达标志
失败后延长上传间隔
成功后恢复默认上传间隔
```

建议策略：

| 场景 | 处理方式 |
| --- | --- |
| Wi-Fi 未连接 | 跳过上传，只保留重连和心跳日志 |
| HTTP 超时 | 记录失败次数，下周期重试 |
| 连续失败 | 临时拉长上传周期 |
| 服务恢复 | 重置失败次数，恢复正常周期 |

### 6.2 ESP32 响应解析优化

当前上传响应主要通过字符串匹配判断：

```text
"alarm":true
```

建议后续改为 JSON 解析，推荐使用 ESP-IDF 常见的 `cJSON`：

```json
{
  "code": 0,
  "message": "upload accepted",
  "alarm": true,
  "alarm_level": "high",
  "alarm_action": "beep_led",
  "event_id": 12
}
```

优化后 ESP32 可根据字段执行不同动作：

| 字段 | 用途 |
| --- | --- |
| alarm | 是否触发本地告警 |
| alarm_action | 控制 LED、蜂鸣器或其他外设 |
| event_id | 记录本次告警对应服务端事件 |
| message | 打印调试日志 |

### 6.3 资源监控与降级

ESP32-S3-CAM 资源有限，应持续关注：

```text
free_heap
PSRAM 使用情况
JPEG 图片大小
上传耗时
摄像头采集失败次数
HTTP 失败次数
```

可设计降级策略：

| 条件 | 动作 |
| --- | --- |
| free_heap 过低 | 延长上传周期 |
| 上传连续超时 | 降低上传频率 |
| 图片过大 | 提高 JPEG 压缩率 |
| 摄像头采集失败 | 延迟重试或重新初始化摄像头 |

### 6.4 DMA 双缓存建议

当前项目不建议优先启用应用层双缓存。`esp32-camera` 驱动底层已经负责摄像头 DMA 与帧缓冲管理，当前使用单帧缓冲：

```text
fb_count = 1
grab_mode = CAMERA_GRAB_WHEN_EMPTY
fb_location = CAMERA_FB_IN_PSRAM
```

对于周期性上传 JPEG 图片的场景，单缓冲更省内存、更稳定、更容易调试。只有在需要提升采集频率、上传和采集并行、PSRAM 余量充足时，才建议评估：

```text
fb_count = 2
```

文档或简历中可以表述为：

```text
基于 esp32-camera 驱动完成 OV3660 JPEG 采集，帧缓冲分配至 PSRAM；当前采用单帧缓冲保证稳定性，并预留双帧缓冲流水化采集优化方案。
```

## 7. 接口安全优化

### 7.1 API Token 鉴权

当前 ESP32 端已经支持通过 `X-API-Token` 请求头发送 Token，Linux 配置中也已有 `API_TOKEN`。后续建议在 FastAPI 端补齐鉴权逻辑：

```text
未配置 API_TOKEN：开发环境允许访问
配置 API_TOKEN：所有 ESP32 接口必须携带正确 X-API-Token
```

需要保护的接口：

```text
POST /api/upload
POST /api/heartbeat
POST /api/alarms/{event_id}/handle
```

可选保护的接口：

```text
GET /api/devices
GET /api/alarms
GET /api/latest
```

### 7.2 请求参数校验

建议进一步校验：

| 参数 | 校验方式 |
| --- | --- |
| device_id | 长度限制、字符白名单 |
| image | JPEG 魔数、大小限制、content-type |
| rssi | 合理范围，例如 -120 到 0 |
| free_heap | 非负整数 |

当前已有 JPEG 格式与大小校验，后续可补齐设备字段校验。

## 8. 图片存储优化

### 8.1 按日期和设备分目录

当前图片保存在 `data/images/raw` 和 `data/images/result` 下。多设备长期运行后，单目录文件会越来越多。建议改为：

```text
data/images/raw/YYYY-MM-DD/device_id/xxx.jpg
data/images/result/YYYY-MM-DD/device_id/xxx_result.jpg
```

优势：

- 文件更易管理。
- 便于按日期归档。
- 便于按设备排查。
- 后续清理策略更简单。

### 8.2 图片清理策略

建议增加定期清理策略：

| 图片类型 | 建议策略 |
| --- | --- |
| 普通检测图片 | 保留最近 N 天 |
| 告警图片 | 保留更长时间 |
| 失败图片 | 保留短时间用于排查 |

建议配置项：

```yaml
NORMAL_IMAGE_RETENTION_DAYS: 7
ALARM_IMAGE_RETENTION_DAYS: 30
```

## 12. 推荐实施顺序

建议按以下顺序推进：

```text
第一步：补齐 Linux API Token 鉴权
第二步：增强多设备状态字段与在线离线判断
第三步：优化 Web 页面设备列表和事件筛选
第四步：ESP32 上传响应改为 JSON 解析
第五步：增加 ESP32 上传失败统计与退避策略
第六步：图片按日期和设备分目录保存
第七步：增加 Linux 端关键耗时日志
第八步：评估是否引入异步推理队列
第九步：预留 MQTT/WebSocket 控制通道扩展
```

优先级划分：

| 优先级 | 优化项 |
| --- | --- |
| P0 | API Token 鉴权、多设备状态增强、Web 设备列表 |
| P1 | ESP32 JSON 响应解析、上传失败退避、图片目录优化 |
| P2 | 日志耗时统计、数据库字段增强、在线离线判断 |
| P3 | 异步推理队列、MQTT/WebSocket 控制通道 |
| P4 | TCP 长连接实验、双帧缓冲采集优化 |

## 14. 本阶段不纳入的优化

根据当前项目目标，以下内容暂不作为本阶段优化重点：

- 告警规则增强。
- AI 模型训练、模型精度提升和检测效果优化。
- 自定义火焰、烟雾或危险物品模型训练。
- 复杂视频流服务。
- 公网云平台接入。
- 大规模分布式推理。

这些内容可以作为后续扩展方向，但当前阶段更建议先把工程主链路、多设备管理和稳定性优化做好。
