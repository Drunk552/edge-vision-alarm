# 开发过程知识点

## 第一阶段：Linux 服务端基础链路

### 阶段目标

先不接入 YOLOv8 和 ESP32 硬件，优先跑通 Linux 服务端基础链路：

```text
HTTP 请求 -> FastAPI 接口 -> 图片校验 -> 图片保存 -> SQLite 记录 -> JSON 响应
```

当前已完成：

- `GET /api/health` 健康检查接口。
- `POST /api/upload` 图片上传接口。
- `POST /api/heartbeat` 设备心跳接口。
- JPEG 图片保存到 `linux/data/images/raw/`。
- 上传事件写入 `alarm_events`。
- 设备状态写入 `device_status`。

### 核心开发步骤

1. 建立 `linux/` 服务端工程结构，按 `api`、`services`、`repositories`、`core` 分层。
2. 使用 `config.yaml` 管理端口、数据库路径、图片目录、模型路径等配置。
3. 使用 SQLite 初始化基础数据表：`alarm_events`、`device_status`、`detection_targets`。
4. 实现 `/api/health`，用于检查服务、数据库和图片目录状态。
5. 实现 `/api/upload`，接收 `device_id`、`image`、`rssi`、`free_heap`。
6. 对上传图片进行 JPEG 格式校验和大小限制。
7. 将图片保存到本地磁盘，数据库只保存图片路径。
8. 上传成功后返回统一 JSON，预留 `alarm`、`targets`、`event_id` 字段。
9. 使用 `pytest`、`ruff` 验证功能和代码质量。
10. 使用浏览器和 `curl` 进行真实 HTTP 接口测试。

### 关键知识点

- **FastAPI**：用于搭建轻量 HTTP 服务，适合实现 ESP32 上传图片、心跳上报和 Web 查询接口。
- **分层架构**：`api` 只处理 HTTP，`services` 编排业务流程，`repositories` 负责数据库读写，避免代码混在一起。
- **SQLite**：适合课程设计和边缘端原型，不需要单独部署数据库服务。
- **图片存储方式**：图片保存到磁盘，SQLite 只保存路径，避免数据库体积过大，也方便 Web 页面展示。
- **统一响应结构**：接口统一返回 `code`、`message` 和业务字段，便于 ESP32 和 Web 端解析。
- **健康检查接口**：用于快速判断服务是否正常，包含数据库、图片目录、模型文件等状态。

### 开发中遇到的问题

- 访问 `http://localhost:8000/` 返回 `404 Not Found` 是正常现象，因为当前没有开发首页路由，应访问 `/api/health`。
- `model_loaded=false` 是正常现象，因为第一阶段还未接入 YOLOv8 模型。
- `curl` 上传文件时，`image=@xxx.jpg;type=image/jpeg` 需要加引号，否则分号会被 shell 当成命令分隔符。
- 测试阶段发现新版 `FastAPI/TestClient` 依赖组合在当前环境下不稳定，因此单元测试改为直接测试接口函数和服务层逻辑。

## 第二阶段：数据查询接口

### 阶段目标

在第一阶段“写入数据”的基础上，补齐服务端“查询数据”的能力：

```text
SQLite 事件/设备数据 -> Repository 查询 -> FastAPI 查询接口 -> JSON 响应
```

当前已完成：

- `GET /api/latest` 查询最新一次上传/检测事件。
- `GET /api/alarms` 分页查询历史事件。
- `GET /api/devices` 分页查询设备状态。

### 核心开发步骤

1. 在 `alarm_repository.py` 中增加事件查询方法：`get_latest_event()`、`list_events()`。
2. 在 `device_repository.py` 中增加设备查询方法：`list_devices()`。
3. 新增 `query_api.py`，统一管理查询类 HTTP 接口。
4. 在 `main.py` 中注册查询路由。
5. 为查询接口补充分页参数校验，限制 `page` 和 `page_size`。
6. 编写查询接口测试，验证最新事件、历史事件和设备状态都能查回。

### 关键知识点

- **Repository 查询封装**：SQL 查询集中放在数据层，接口层不直接拼接 SQL。
- **分页查询**：列表接口使用 `page`、`page_size`、`total`、`items`，方便后续 Web 页面分页展示。
- **最新结果接口**：`/api/latest` 是 Web 首页和调试时最常用的接口，用于快速查看最近一次上传/检测结果。
- **设备状态接口**：`/api/devices` 用于查看 ESP32 是否在线、RSSI 和剩余内存等运行状态。

### 开发中遇到的问题

- 直接调用 FastAPI 路由函数测试时，`Query(default=None)` 会作为对象传入，导致 SQLite 参数绑定失败。处理方式是改为普通默认值，并在函数内部手动校验。
- 启动服务时报 `address already in use`，原因是 8000 端口已有旧的 `uvicorn` 进程占用。可用 `lsof -i :8000` 查看，并用 `kill -9 <PID>` 释放端口。

## 第三阶段：YOLOv8 检测链路

### 阶段目标

将上传链路升级为视觉检测链路：

```text
图片上传 -> 保存原图 -> YOLOv8 推理 -> 告警规则判断 -> 结果入库 -> JSON 响应
```

当前已完成：

- 新增 `BaseDetector` 抽象接口。
- 新增 `Yolov8Detector`，用于加载 `yolov8n.pt` 并执行检测。
- 新增 `EmptyDetector`，在模型或依赖不可用时保持上传链路可运行。
- 新增 `AlarmRuleEngine`，当前规则为检测到 `person` 且置信度达标则告警。
- 新增 `TargetRepository`，将检测目标写入 `detection_targets`。
- 使用 OpenCV 绘制检测框并保存结果图。

### 关键知识点

- **检测器抽象**：业务层只依赖 `detect(image_path)`，后续替换 YOLO、ONNX 或其他模型时不影响接口层。
- **模型文件管理**：`yolov8n.pt` 放在 `linux/models/`，不提交 Git。
- **结果图生成**：原图保存在 `data/images/raw/`，带检测框图片保存在 `data/images/result/`。
- **告警规则独立**：检测模块只输出目标，是否告警由规则引擎判断，避免模型逻辑和业务规则混在一起。

### 开发中遇到的问题

- 清华 PyPI 镜像下载 `ultralytics` 出现 403，可切换阿里云、华为云等镜像。
- `opencv-python` 包较大，服务器环境后续可考虑改为 `opencv-python-headless`。
- 单元测试中的假 JPEG 只有文件头尾，不能被 YOLO/OpenCV 解码。接入真实模型后，旧上传测试会误触发 YOLO 推理并失败。处理方式是测试上传链路时显式注入 `EmptyDetector()`，检测链路单独用假检测器测试。

### 当前验证结果

```text

python3 -m ruff check app tests
All checks passed

python3 -m ruff format --check app tests
36 files already formatted
```

### 简历表达参考

```text
接入 Ultralytics YOLOv8 检测模块，设计 Detector 抽象层解耦模型推理与业务流程，并结合告警规则引擎实现人员目标检测、告警状态判断、检测目标入库和结果图保存。
```

## 第五阶段：ESP32 Wi-Fi 联网

### 阶段目标

在 ESP32-S3 最小固件基础上，先跑通 Wi-Fi STA 联网能力：

```text
ESP32-S3 启动 -> 初始化 NVS -> 初始化网络协议栈 -> 连接 Wi-Fi -> 输出 IP 和 RSSI
```

当前已完成：

- 新增 `components/wifi` 独立组件。
- 通过 `menuconfig/sdkconfig` 配置 Wi-Fi SSID 和密码。
- 初始化 NVS、`esp_netif` 和 ESP-IDF 默认事件循环。
- 初始化 Wi-Fi STA 模式并连接指定热点。
- 获取并打印 IP 地址和 RSSI。
- Wi-Fi 断开后自动重连。

### 关键知识点

- **ESP-IDF 组件化结构**：`main` 只负责入口装配，Wi-Fi 逻辑拆到 `components/wifi`，符合模块职责边界。
- **NVS 初始化**：Wi-Fi 驱动依赖 NVS 保存射频校准和配置数据，首次启动可能出现重新校准日志。
- **Wi-Fi STA 模式**：ESP32 作为客户端连接路由器或电脑热点，适合后续访问 Linux 服务端接口。
- **事件驱动联网**：通过 `WIFI_EVENT` 和 `IP_EVENT` 处理启动、断开、重连和获取 IP。
- **敏感配置管理**：SSID 和密码只写入本地 `sdkconfig`，不写入源码和 `sdkconfig.defaults`。

### 开发中遇到的问题

- 初始实现把 `wifi_manager.c/.h` 放在 `main` 目录，不符合开发文档要求，已调整到 `esp32/components/wifi/`。
- `main` 组件显式声明依赖后，需要补充 `spi_flash` 依赖，否则 `esp_flash.h` 无法编译。
- 首次烧录日志中出现 RF calibration fallback 属于常见校准信息，不影响 Wi-Fi 联网。

### 当前验证结果

```text

Project build complete
```

真实硬件验证：

- ESP32-S3 成功连接电脑热点。
- 串口日志显示 IP：`192.168.137.121`。
- 串口日志显示 RSSI 约为 `-29 dBm` 到 `-34 dBm`。
- 系统心跳持续输出 `Wi-Fi connected`。

### 简历表达参考

```text
基于 ESP-IDF 实现 ESP32-S3 Wi-Fi STA 联网模块，采用组件化结构拆分网络逻辑，完成 NVS、网络协议栈、事件循环和断线重连处理，并通过串口日志输出 IP、RSSI 和运行状态。
```

## 第六阶段：ESP32 HTTP 心跳上报

### 阶段目标

在 Wi-Fi 联网成功的基础上，验证 ESP32 与 Linux 服务端的真实 HTTP 通信：

```text
ESP32-S3 -> HTTP POST -> Linux /api/heartbeat
```

当前已完成：

- 新增 `components/config` 配置组件。
- 新增 `components/heartbeat` 心跳组件。
- 通过 ESP-IDF HTTP Client 发送 JSON 心跳请求。
- 上报 `device_id`、`rssi`、`free_heap` 和 `status`。
- Wi-Fi 未连接时跳过心跳请求。

### 关键知识点

- **配置集中管理**：设备 ID、服务端地址、心跳周期和 HTTP 超时统一从配置组件读取，避免散落在业务代码中。
- **HTTP Client**：ESP32 使用 `esp_http_client` 发送 `application/json` 请求，适合对接 FastAPI 接口。
- **心跳任务**：心跳作为独立 FreeRTOS 任务周期运行，不阻塞主入口和 Wi-Fi 事件处理。
- **网络状态检查**：发送请求前先判断 Wi-Fi 是否已连接，降低无效请求和错误日志。
- **敏感信息处理**：服务端地址和可选 Token 走本地 `sdkconfig`，不写入源码。

### 开发中遇到的问题

- 阶段八只实现 `/api/heartbeat`，不提前实现图片上传，避免跳阶段开发。
- `main` 只启动心跳任务，HTTP 请求构造和发送逻辑放在 `heartbeat` 组件中。
- 服务端运行在电脑热点环境时，ESP32 访问地址通常需要使用电脑热点网关 IP。

### 当前验证结果

```text

Project build complete
```

真实联调验证：

- ESP32 调用 `/api/heartbeat`：验证成功。
- Linux `/api/devices` 显示真实设备：验证成功。
- Web 页面设备状态显示最新 RSSI、Heap 和最近在线时间。

### 简历表达参考


## 阶段九：ESP32 摄像头初始化知识点

### 本阶段目标

ESP32-S3-CAM 初始化 OV3660 摄像头，使用 JPEG 格式采集图像帧，并验证帧缓冲申请和释放流程。

### 关键知识点

- **摄像头组件化**：摄像头初始化和采集逻辑放在 `components/camera`，`main` 只负责入口装配。
- **DVP 引脚配置**：XCLK、SCCB、VSYNC、HREF、PCLK 和 D0-D7 必须与实际开发板一致。
- **PSRAM 帧缓冲**：摄像头 JPEG 帧缓冲放入 PSRAM，避免占用内部 RAM。
- **PSRAM 模式**：当前 ESP32-S3 板载 8MB Embedded PSRAM 使用 Octal 模式。
- **资源释放**：每次获取 `camera_fb_t` 后必须调用释放接口，避免帧缓冲泄漏。

### 验证结果

- 串口识别到 OV3660 摄像头。
- PSRAM 初始化成功。
- 摄像头初始化成功，日志显示 `camera init success`。

## 阶段十：ESP32 图片上传知识点

### 本阶段目标

ESP32 拍摄 JPEG 图片，通过 HTTP `multipart/form-data` 上传到 Linux `/api/upload` 接口。

### 关键知识点

- **HTTP 上传组件化**：图片上传逻辑放在 `components/http`，不放入摄像头组件或 `main`。
- **multipart/form-data**：上传请求包含 `device_id`、`image`、`rssi`、`free_heap` 字段。
- **网络状态检查**：上传前先判断 Wi-Fi 是否已连接，未联网时跳过上传。
- **帧缓冲生命周期**：上传成功或失败后都必须释放摄像头帧缓冲。
- **任务并行运行**：心跳任务和图片上传任务独立运行，互不阻塞。

### 验证结果

- ESP32 串口显示 `image uploaded, status=200`。
- Linux `/api/upload` 接收真实 ESP32 图片成功。
- 心跳上报与图片上传可以同时工作。

## 阶段十一：ESP32 告警联动知识点

### 本阶段目标

ESP32 解析 Linux `/api/upload` 返回的 `alarm` 字段，在 `alarm=true` 时触发本地板载 LED 闪烁。

### 关键知识点

- **告警组件化**：本地告警逻辑放在 `components/alarm`，不写入 `main` 或 HTTP 上传细节中。
- **响应解析**：ESP32 上传图片后读取响应体，只解析关键字段 `alarm=true`。
- **GPIO 联动**：当前使用 GPIO38 作为板载 LED 告警引脚，默认高电平点亮。
- **队列触发**：HTTP 上传任务只投递告警事件，LED 闪烁由 `alarm_task` 执行。
- **硬件范围控制**：当前未接蜂鸣器和舵机，只完成板载 LED 告警闭环。

### 验证结果

- 使用 Linux 临时强制告警规则返回 `alarm=true`。
- ESP32 串口显示 `server returned alarm=true`。
- ESP32 串口显示 `local LED alarm triggered`、`local LED alarm started`、`local LED alarm finished`。
- 测试完成后已恢复 Linux 正常告警规则。

## 阶段十二：系统联调与稳定性测试知识点

### 本阶段目标

验证 ESP32、Linux 服务端、YOLOv8 检测链路和 Web 页面能形成完整闭环，并检查常见异常场景下系统是否有明确日志。

### 关键知识点

- **闭环验证顺序**：先验证 Linux 接口，再验证 ESP32 编译，最后进行实机 Wi-Fi、心跳、拍照、上传、告警和 Web 展示联调。

- **稳定性观察指标**：重点观察 ESP32 free heap、RSSI、上传状态码、服务端请求日志和 Web 页面刷新情况。
- **模型缺失降级**：YOLOv8 模型不存在时，Linux 端降级为空检测器，保证上传链路不中断。
- **容量风险**：ESP32 app 分区剩余空间约 5%，后续新增功能前需关注分区表和固件体积。

- Linux `black` 逐文件格式检查通过。
